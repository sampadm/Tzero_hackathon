Section 01
Problem Statement

Tokenizing real-world assets requires translating complex legal documents — like equity term sheets — into secure, on-chain smart contracts. Today, this translation is entirely manual, demanding rare and expensive cross-domain expertise (legal + blockchain), drastically slowing time-to-market, and introducing high risk of human error when converting nuanced legal conditions into immutable code.

Private company asset owners and their intermediaries (law firms, placement agents, transfer agents) have no self-service path to bring assets onto Tzero's platform. Every tokenization engagement is custom, slow, and costly — limiting Tzero's ability to scale its asset base.

BYOA solves this by introducing an AI-driven onboarding workflow on Tzero.com that accepts standardized NVCA-format term sheets, extracts key financial parameters via AI, automatically generates compliant ERC-3643 smart contract code, routes through a dual-approval gate, and triggers a listing request upon successful deployment.

Section 03
Personas & Key User Journeys
Persona 1 — The Intermediary
A professional at a law firm, placement agent, or transfer agent acting on behalf of a private company asset owner.
Primary User
GoalEfficiently onboard a client's equity asset onto Tzero with minimal back-and-forth
Technical LevelHigh — legal & financial. Limited blockchain expertise.
Pain TodayManual process is slow, opaque, requires deep Tzero coordination
1
Registration & Firm Onboarding
  • Intermediary firm receives invite or applies for BYOA access
  • Firm-level account created on Tzero.com
  • Firm completes KYB (Know Your Business) verification
  • Account activated — single firm login with BYOA dashboard access
2
Asset Submission Initiation
  • Logs in → navigates to BYOA section on Tzero.com
  • Clicks "Onboard New Asset"
  • Fills in: asset name, issuing company, asset type (Equity), estimated valuation, jurisdiction (US)
3
Term Sheet Upload
  • Uploads NVCA-format PDF term sheet
  • System validates file format and size
  • AI pipeline processes document (async — status indicator shown)
  • Extraction results displayed with confidence scores per field (HIGH / MEDIUM / LOW)
  • LOW-confidence fields flagged for mandatory review
4
Intermediary Review & Override
  • Reviews all extracted fields in structured form UI
  • Can edit any field — overrides logged with timestamp and user
  • Low-confidence fields highlighted — must explicitly confirm or correct each
  • Once all fields confirmed → submits for Tzero compliance review
5
Tzero Compliance Review (async)
  • Tzero internal reviewer receives notification
  • Reviews extracted parameters + intermediary overrides
  • Can: Approve / Request Changes / Reject with reason
  • If changes requested → intermediary notified, returns to step 4 with comments
6
Smart Contract Generation
  • System auto-generates ERC-3643 compliant smart contract code
  • Human-readable contract summary generated alongside code
  • Both displayed to intermediary for final review
  • Intermediary confirms match with client's intent → approves
  • Tzero compliance does final sign-off on generated contract
7
Deployment
  • Tzero deploys contract to Ethereum mainnet
  • Transaction hash returned — visible in dashboard
  • Asset status updated: "Deployed — Pending Listing Activation"
  • Listing request automatically triggered to Tzero's trading team
8
Post-Deployment
  • Intermediary receives: contract address, transaction hash, human-readable summary PDF, fee invoice
  • Tzero trading team activates listing separately (out of BYOA scope)
Persona 2 — Tzero Compliance Reviewer
Tzero internal legal/compliance team member responsible for reviewing submissions before deployment.
Internal User
GoalQuickly verify extracted parameters are accurate and compliant (Reg D/Reg S)
Pain TodayNo structured tooling — reviews are ad hoc via email/docs
1
Notification
  • Receives in-platform notification + email when intermediary submits for review
2
Review Dashboard
  • Sees queue of pending submissions
  • Opens submission → views original PDF side-by-side with AI-extracted fields
  • Intermediary overrides highlighted in diff view
  • Compliance flags raised by AI (Reg D/Reg S, accredited investor gate)
3
Decision
  • Approve → advances to contract generation
  • Request Changes → adds per-field comments → notifies intermediary
  • Reject → adds reason → notifies intermediary → case closed
4
Contract Sign-Off (second gate)
  • After contract is generated, does final review of Solidity code + summary
  • Approves → Tzero deploys to Ethereum mainnet

Section 06
Technical Considerations
AI Pipeline
  • Document ParsingPDF text extraction with layout awareness — handles multi-column NVCA formats. Libraries: pdfplumber (open source) + LLM, or AWS Textract / Google Document AI.
  • Parameter ExtractionLLM-based extraction (Claude API or GPT-4) with structured output schema mapping to NVCA equity term sheet fields. Prompt-engineered for NVCA-standard documents.
  • Confidence ScoringPer-field confidence score based on extraction certainty. Fields below threshold flagged for human review. Threshold tunable post-launch.
  • Compliance DetectionRule-based layer on top of extraction — flags Reg D / Reg S applicability based on investor type, geography, and offering size fields.
Smart Contract Generation
  • StandardERC-3643 (T-REX protocol) — permissioned token standard designed for regulated securities.
  • ApproachTemplate-based generation — pre-audited ERC-3643 contract templates parameterized by extracted fields. Reduces risk vs. fully generative code. Templates cover: standard equity, preferred equity with liquidation preference, equity with conversion rights.
  • Audit RequirementAll templates must be pre-audited by a third-party smart contract auditor before MVP launch.
  • OutputSolidity source code + ABI + deployment bytecode.
Deployment Infrastructure
  • WalletTzero-controlled deployment wallet (multisig recommended).
  • Trigger GateDeployment triggered only after dual approval is recorded in audit log.
  • MonitoringTransaction monitoring — confirm on-chain inclusion before marking as deployed. Gas management handled by Tzero, not exposed to intermediary.
Data & Security
  • EncryptionAll uploaded PDFs and extracted data encrypted at rest (AES-256) and in transit (TLS 1.3).
  • Data IsolationTerm sheet data and generated contracts never exposed to third parties. Firm accounts can only see their own submissions.
  • Audit LogImmutable — append-only, tamper-evident. Role-based access enforced throughout.

Section 07
UI Style Preferences

Must strictly conform to Tzero.com's existing design system — typography, color palette, component library, spacing, and interaction patterns. BYOA is a new section/module within Tzero.com — accessible via authenticated navigation.

  • →
    Progressive Disclosure
    Multi-step wizard pattern, one clear action per screen. Users are never overwhelmed by the full workflow at once.
  • ◎
    Status Transparency
    Always-visible submission status (e.g., "In Review," "Changes Requested," "Deployed") with timestamps. No opaque waiting states.
  • ▣
    Confidence Visualization
    Color-coded field confidence (green/yellow/red) in the extraction review UI — informative without being visually overwhelming.
  • ⊞
    Side-by-Side Review
    Original PDF viewable alongside extracted fields during review step. Enables fast cross-referencing without tab-switching.
  • ≡
    Audit Trail Visibility
    Intermediary can see a full timeline of all actions taken on their submission — overrides, approvals, rejections, deployments.
Section 08
Corner Cases
AI Extraction
ScenarioHandling
PDF is scanned image (not text-based)Reject with error: "Please upload a text-based PDF. Scanned images are not supported."
Term sheet is missing critical fields (e.g., valuation cap absent)Flag missing required fields — block progression until intermediary manually enters value
Ambiguous or contradictory clausesAI flags the specific clause, marks field as LOW confidence, requires intermediary review
Term sheet references external documents ("as defined in Schedule A")AI flags dependency — intermediary prompted to manually enter the referenced value
PDF contains multiple term sheets (multi-asset upload)Reject with error — one term sheet per submission for MVP
Dual Approval Gate
ScenarioHandling
Tzero compliance requests changes after intermediary already approvedIntermediary notified, returns to review step — prior approval invalidated, must re-approve after changes
Tzero compliance rejects submissionIntermediary notified with reason — submission closed, new submission required
Intermediary abandons workflow mid-stepDraft state saved — intermediary can resume from last completed step
Intermediary wants to withdraw before Tzero reviewAllow withdrawal — submission cancelled, audit log updated
Deployment
ScenarioHandling
Ethereum network congestion causes deployment failureRetry logic with exponential backoff — intermediary notified if unresolved after 3 attempts, Tzero ops alerted
Gas price spike increases deployment cost significantlyTzero absorbs gas cost (not passed to intermediary in MVP) — ops team alerted for high-gas events
Contract deployment succeeds but transaction not confirmed within timeoutMonitor mempool — alert Tzero ops team, do not mark as deployed until confirmed
Duplicate submission (same asset submitted twice)Detect by company name + asset name combination — warn intermediary of potential duplicate before submission
Compliance
ScenarioHandling
Asset valuation entered is implausibly low or highSoft warning to intermediary — does not block, but logged for Tzero compliance reviewer attention
Term sheet indicates non-accredited investorsAI flags as non-compliant with Reg D — Tzero compliance reviewer alerted, submission blocked from auto-advancing
Issuing company is foreign-incorporated (non-US)Flagged for Tzero compliance review — may require Reg S treatment; reviewer decides
Account & Access
ScenarioHandling
Firm account not yet KYB-verified tries to submitBlocked — prompt to complete KYB before submitting assets
Tzero compliance reviewer is unavailable (OOO)No automated escalation in MVP — ops team manages reviewer coverage manually
